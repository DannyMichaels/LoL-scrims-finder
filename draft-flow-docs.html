Draft Tool Full Flow

Real-time champion draft system for the LoL Scrims Finder app. Supports Captain Pick and Individual Pick/Ban modes with Fearless draft for best-of series.

Overview

The draft tool lets two teams ban and pick champions in real-time, following the official League of Legends tournament format. It can be used standalone (navigate to /draft/new) or linked to a scrim (auto-populates rosters).

Key principle: Server-authoritative The server is the single source of truth. All actions are validated server-side before being broadcast. Clients display state, they never compute it independently. This prevents cheating and desync.

Architecture

Three layers work together:

React Frontend (Client)

Zustand store + socket hooks + MUI components. Renders the draft board, champion grid, timer. Emits socket events for user actions. Computes countdown locally from server-provided timerExpiresAt.

Socket.io Layer (Real-time)

Handles all draft events: join, ready, action, swap. Each draft gets a socket room (draft_{id}). Server broadcasts state changes to all connected clients.

Draft Engine (Server)

State machine in draftEngine.services.js. Validates actions, advances the action pointer, manages auto-complete on timeout. Reads/writes MongoDB via Mongoose.

1. Create Draft

A user navigates to /draft/new and fills the form:

User fills DraftCreateForm

Selects mode (Captain/Individual), enters team names, sets timer (15-90s), optionally enables Fearless mode + series info.

POST /api/drafts

Client calls createDraft() service. Server runs draftEngine.createDraft() which pre-generates all 20 actions with type/team/phase, saves to MongoDB.

Redirect to /draft/:id

User is navigated to the draft page. The DraftPage component fetches initial state via GET /api/drafts/:id and connects the socket.

Pre-generated Actions

At creation time, the engine builds an array of 20 action objects. Each has: actionIndex, type (ban/pick), team (blue/red), phase. The draft has a currentActionIndex pointer that starts at 0 and increments after each action completes.

2. Join & Ready

DraftJoinModal appears

First visitor sees "Choose Your Side" — picks Blue, Red, or Spectate. Selection is stored in the Zustand store (myTeam).

Socket: draft:join

Client emits draft:join { draftId, team }. Server adds socket to room draft_{id}, sends back draft:state with full draft object.

Ready button

Each team captain clicks "Ready". Client emits draft:ready { draftId, team }.

Both ready → Draft starts

When both blueTeam.ready and redTeam.ready are true, server sets status: "in_progress", broadcasts full state, and starts the first action timer.

3. Drafting Phase

The core loop for each of the 20 actions:

Server starts timer

Emits draft:timer_start { expiresAt }. Client stores timerExpiresAt and the useDraftTimer hook counts down locally every second.

Active team selects a champion

Player clicks a champion in the ChampionGrid → sets selectedChampion in store. Clicking "Lock In" emits draft:action { draftId, championId, championName }.

Server validates

draftEngine.validateAction() checks: correct team's turn, champion not already used, not fearless-locked. Rejects with draft:error if invalid.

Server applies action

draftEngine.applyAction() fills the action's championId, completedAt, increments currentActionIndex. For picks, also assigns champion to the player's slot.

Broadcast to all clients

Emits draft:action_update { action, currentActionIndex } to the room. If the phase changed, also emits draft:phase_change. Then starts the next timer.

4. Timer & Auto-Complete

The server runs a setTimeout for each action. Timers are tracked in a Map<draftId, timerId>.

Timer sync strategy Server stores timerExpiresAt as an absolute timestamp. Clients compute remaining seconds locally: Math.ceil((expiresAt - Date.now()) / 1000). This avoids drift from network latency.

On timeout:

  • Ban action: auto-completes with null champion (no ban used)
  • Pick action: auto-completes with a random available champion from the cached champion list

Auto-completed actions are flagged with wasAutoCompleted: true.

5. Swap Phase

After all 20 actions complete, the draft enters a 30-second swap phase:

draft:swap_phase_start

Server sets status: "swap_phase", swapPhaseExpiresAt. Clients show swap UI.

Players request swaps

Teammate clicks another teammate's pick to propose a trade. Emits draft:swap_request { fromSlot, toSlot, team }.

Target accepts/rejects

Emits draft:swap_respond { swapRequestId, accept }. If accepted, server swaps the champion fields on both player slots.

Phase expires

After 30s, server auto-completes the draft: status: "completed", emits draft:completed.

6. Completion

Draft is now status: "completed". The final state is stored in MongoDB. If linked to a scrim, the scrim can reference the draft via _scrim on the draft document. The UI shows the final picks/bans with a "Draft Complete" header.

Captain Mode

One drafter per team controls all bans and picks. The 20 actions are fully sequential — one team acts, then the other, following the tournament snake pattern.

Captain picks fill slots in order: First pick goes to slot 0, second to slot 1, etc. The captain chooses the champion, but it gets assigned to their team's players sequentially.

Individual Mode

Each player bans and picks for themselves.

Simultaneous bans Ban phases are different in Individual mode. All players submit bans at the same time (hidden from opponents). Once all bans are in (or timer expires), they're revealed together. This matches the League client experience.

How simultaneous bans work internally:

  1. Server calls initSimultaneousBanPhase() — sets expectedBanCount (6 for phase 1, 4 for phase 2)
  2. Each player's ban is stored in draft.pendingBans Map (hidden from other team)
  3. When pendingBans.size >= expectedBanCount, all bans are revealed at once
  4. Server emits draft:action_update with revealedBans array
  5. Picks remain sequential (B-RR-BB-R / R-BB-R) with specific playerSlot assignments

Fearless Mode

For best-of series. Champions picked in earlier games become locked in later games.

ModeLock RuleExample
Soft Locked for the team that picked it Blue picked Jinx in Game 1 → Blue can't pick Jinx in Game 2, but Red still can
Hard Locked for both teams Anyone picked Jinx in Game 1 → neither team can pick Jinx in Game 2

How it works:

  • Drafts in a series share a seriesId and increment gameNumber
  • When creating Game 2+, the engine queries previous completed drafts in the same series
  • Populates previouslyPickedChampions array on the new draft
  • During validation, isFearlessLocked() checks if the champion is locked for the acting team
  • UI shows locked champions with a lock icon in the champion grid
